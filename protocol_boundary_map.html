Six protocol families, six distinct layers of the agent interaction stack. Each owns a problem space and hands off to adjacent layers. Click any layer to expand scope and handoffs.

Coordination Layer DMSC
Cross-organizational team assembly · gateway orchestration · cross-domain governance policy

Owns

  • Multi-agent team assembly and task delegation across admin domains
  • Gateway-mediated capability negotiation and semantic intent routing
  • Governance policy enforcement for regulated environments (financial, telco)
  • Provenance receipts and session audit records

Hands off to

  • Transport layer (HTTP, AGTP) for actual message delivery
  • OAuth for token-level authorization within delegated tasks
  • DAWN for discovering agents before gateway routing

Transport & Session Layer AgentProtocol / A2A
Agent-to-agent and agent-to-resource communication · session lifecycle · method vocabulary

Owns

  • Two competing designs: AGTP (new protocol, port 4480, 18-method vocabulary) and A2A (HTTP-based, no IETF-filed normative docs from Google)
  • Session establishment and lifecycle
  • Protocol-level authority scope declaration (AGTP Agent-Authority-Scope header)
  • Agent identity binding to TLS via X.509 extension (AGTP-CERT)

Hands off to

  • DAWN to discover endpoints before connecting
  • WIMSE or WebBotAuth for cryptographic identity binding
  • OAuth for the authorization tokens carried in the session

Discovery Layer DAWN
Finding agents, workloads, and services before any connection or authentication begins

Owns

  • Protocol-neutral discovery of agents, workloads, and named entities across admin boundaries
  • Capability advertisement and search
  • Authentication and authorization of discovered entities are explicitly out of scope
  • IETF 126 (Bangkok) has DAWN BoF time — WG formation imminent

Hands off to

  • AgentProtocol (AGTP, A2A, HTTP) for the actual connection
  • WIMSE or WebBotAuth to authenticate the discovered entity
  • OAuth for authorization of the request to the discovered entity

Web-Facing Auth Layer WebBotAuth
Authenticating bots and AI agents to public web origin servers — replacing IP allowlisting

Owns

  • Bot/agent authentication to websites built for human users
  • Signature Agent Card: JSON identity document with public keys and capabilities
  • HTTP Message Signatures over the agent's requests
  • Anonymous path (Privacy Pass) as the privacy-preserving alternative

Hands off to

  • OAuth for resource-level access decisions after authentication
  • WIMSE for internal service mesh identity (different trust perimeter)

Workload Identity Layer WIMSE
Cryptographic identity for running agent processes within and across service mesh boundaries

Owns

  • SPIFFE SVIDs and workload identity credentials for the running process
  • Mutual TLS between services using workload certificates
  • Cross-domain credential verification and trust domain discovery
  • Condition-bounded credentials (posture-gated, not time-bounded)

Hands off to

  • OAuth via draft-ietf-oauth-spiffe-client-auth (WG-adopted): WIMSE identity → OAuth client assertion → authorization token
  • WebBotAuth for external web-origin authentication (different perimeter)

Authorization Layer OAuth
Token issuance · delegation chains · scope attenuation · revocation · resource access control

Owns

  • The complete authorization lifecycle — every layer above hands authorization decisions here
  • Token exchange and delegation chains (RFC 8693)
  • Rich authorization requests and scope attenuation (RFC 9396 RAR)
  • Transaction tokens for intra-service request tracing (draft-ietf-oauth-transaction-tokens)
  • Cross-domain identity propagation (draft-ietf-oauth-identity-chaining)

Receives from other layers

  • WIMSE identity credential → client assertion for token request
  • WebBotAuth bot identity → precondition for authorization decisions
  • DAWN discovery → client knows which AS to request tokens from
  • DMSC task context → informs scope and delegation depth

Open Boundary Tensions
Tension 01 · Discovery

DAWN vs. DMSC gateway vs. AGTP DISCOVER

Three families claim the discovery slot. DAWN is a standalone federated protocol layer. DMSC embeds discovery inside the gateway's semantic intent router. AGTP defines a DISCOVER method. These are architecturally incompatible at scale — a global standard can have only one answer.

Tension 02 · Identity Perimeter

WebBotAuth httpsig vs. WIMSE mTLS for the same agent

An agent crawling the public web uses WebBotAuth (HTTP message signatures). The same agent calling internal services uses WIMSE mTLS. Both rely on RFC 9421 HTTP Message Signatures as a common substrate, but the two WGs have not specified how credentials compose or translate across the perimeter.

Tension 03 · Authority Declaration

OAuth scope vs. AGTP Agent-Authority-Scope header

AGTP mandates an Agent-Authority-Scope header declaring the agent's authority at the protocol level — but OAuth access tokens carry the same information in scope or authorization_details claims. AGTP-composition specifies AGTP headers take precedence over adjacent-layer fields, but OAuth-native resource servers don't speak AGTP. No spec resolves precedence when they diverge.

Tension 04 · Governance

DMSC cross-org governance vs. OAuth authorization scope

The dunbar-dmsc-gw-scenarios analysis explicitly names MCP and A2A as insufficient for regulated cross-organizational environments. OAuth also doesn't own cross-org governance orchestration. DMSC fills this gap with its mediation layer — but DMSC and OAuth have no explicit composition profile. Whether OAuth's Complex Delegation recharter covers this space is unresolved.
